# CI/CD の仕組み

このリポジトリでは、変更が次の順番で確認されます。

1. ローカルで変更する
2. `pre-commit` がコミット前の確認をする
3. `GitHub Actions` が PR 上で CI を実行する
4. `main` へマージされたら `GitHub Pages` に公開する

## `CI` とは

`CI` は `Continuous Integration` の略です。日本語では「継続的インテグレーション」と呼ばれます。

意味をやさしく言い換えると、次の考え方です。

- 変更は小さくこまめに入れる
- 入れるたびに自動テストする
- 問題があればすぐ分かるようにする

このリポジトリでは、PR を作ると次が自動で実行されます。

- `pre-commit run --all-files`
- `gitleaks`
- `pnpm audit`
- `Vitest`
- `Vite build`
- `Storybook build`
- `SBOM` 生成
- `Playwright`

## `CD` とは

`CD` は `Continuous Delivery` または `Continuous Deployment` の略です。

この教材では「テストを通ったものを GitHub Pages に自動公開する」ので、CD の入り口を体験できます。

## 各ツールの役割

### `mise`

ツールのバージョンをそろえる役目です。
ローカルと CI の差を減らすために使います。

### `pnpm`

JavaScript の依存関係を入れる道具です。
この教材では、ただインストールできればよいのではなく、「同じものを安全に入れる」ために使っています。

- `pnpm-lock.yaml` で依存関係を固定する
- `minimumReleaseAge` で公開された直後のパッケージを少し待つ
- `pnpm audit` で既知の脆弱性を調べる
- `pnpm sbom` で使っている OSS 一覧を出せる

つまり `pnpm` は、インストール担当であると同時に、依存関係の安全管理にも使える道具です。

### `pre-commit`

コミット前に軽い確認を自動実行します。

- 整形されているか
- シークレットが混ざっていないか
- Lint エラーがないか
- Markdown に壊れた書き方がないか
- 型検査が通るか
- 大きすぎる追加ファイルが混ざっていないか

「保存し忘れ」や「小さな文法ミス」を早く止めるのが得意です。

### `gitleaks`

`gitleaks` は、コードや Git 履歴の中に API キー、パスワード、トークンのような秘密情報が入っていないかを検出するツールです。

このツールが必要な理由は、シークレット漏えいは「直して終わり」ではないからです。

- ファイルから消しても Git の履歴には残ることがある
- いったん push すると、公開範囲が広がることがある
- 漏えいしたキーは使えなくする、つまり失効や再発行まで必要になる

そのため、この教材では次の 3 段階で確認しています。

1. `pre-commit` で staged 差分をすばやく確認する
2. `pnpm run secret-scan` でリポジトリ全体を自分で確認する
3. `GitHub Actions` で同じ考え方の確認をサーバー側でも行う

つまり、`gitleaks` は「セキュリティ事故を早めに止める CI/CD の部品」です。

### `Vitest`

単体テストを動かします。
この教材では、ゲームのロジックを UI と分けてあるので、「敵に当たったら 100 点増える」などをテストできます。

### `Playwright`

ブラウザを自動操作して、実際の画面の流れを確認します。
単体テストだけでは見つけにくい、「ボタンを押したら画面が動くか」を確認する役目です。

### `Storybook`

ボタンやパネルのような UI パーツを、アプリ本体とは別に確認できる道具です。

これを入れる理由は次の通りです。

- 見た目の確認をしやすい
- 小さな部品だけをレビューしやすい
- `build` できるので、壊れていないことを CI でも確認できる

### `GitHub Actions`

GitHub 上で自動実行を担当する仕組みです。
`.github/workflows/` に書いた YAML ファイルを読むと、どのタイミングで何を動かすかが分かります。

この教材では CI の中で、依存関係についても次を確認します。

- `pnpm audit` で高 severity の脆弱性を検出する
- `pnpm sbom` で SBOM を生成して artifact として残す

`SBOM` は `Software Bill of Materials` の略です。
やさしく言うと「このアプリにどんな OSS が入っているかの部品表」です。
あとから調査や監査をしやすくするために使います。

## なぜローカルと CI の両方で確認するのか

答えは役割が違うからです。

- ローカル:
  自分がすぐ直せる
- CI:
  チーム全体へ入る前に止める

ローカルだけだと、うっかりフックを無効化して push することがあります。
CI だけだと、毎回 GitHub へ送るまで結果が分からず遅いです。
特にシークレット漏えいは「push した後に見つかる」だけでも被害が大きくなりやすいので、ローカルと CI の両方で確認する価値があります。
両方あると、速さと安全性のバランスが取りやすくなります。

## フォークしたリポジトリでも動く理由

この教材の CI は、基本的に `Secrets` を使いません。

これは、フォーク PR では通常 `Secrets` にアクセスできないためです。
学習者が自分のフォークで安全に試せるように、最初から Secrets なしで動く構成にしています。
