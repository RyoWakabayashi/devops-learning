# OSS 依存関係の安全な扱い方

このページでは、JavaScript の外部パッケージをどう安全に扱うかを説明します。

新人のうちは「アプリのコード」だけに意識が向きやすいですが、実際には `pnpm` で入れる OSS もアプリの一部です。
そのため、依存関係も CI/CD の対象として管理する必要があります。

## なぜ依存関係の管理が必要か

OSS はとても便利ですが、次のようなリスクがあります。

- 脆弱性が見つかっているバージョンを使ってしまう
- 公開された直後の不安定なバージョンをすぐ取り込んでしまう
- 開発者ごとに違うバージョンが入り、再現しづらくなる
- 後から「どの OSS を使っていたか」が分からなくなる

この教材では、こうした問題に対して「固定する」「待つ」「調べる」「記録する」の 4 つを学べるようにしています。

## 1. `pnpm-lock.yaml` をコミットする

`lockfile` は、実際に入れた依存関係の正確な一覧です。

`package.json` だけだと `^` や `~` によって、別の日にインストールしたとき違うバージョンが入ることがあります。
`pnpm-lock.yaml` をコミットしておけば、ローカルと CI で同じ組み合わせを再現しやすくなります。

この教材で `pnpm install --frozen-lockfile` を使っているのは、その lockfile を勝手に書き換えないためです。

## 2. `minimumReleaseAge` で少し待つ

このリポジトリでは `pnpm-workspace.yaml` に次の設定を入れています。

```yaml
minimumReleaseAge: 20160
minimumReleaseAgeStrict: false
```

`20160` 分は 14 日です。

意味は「公開されたばかりのパッケージを、すぐには新規採用しない」です。
これが必要な理由は、もし不正なパッケージ公開や重大な不具合があっても、短い待機期間があると気づける可能性が上がるからです。

この教材では `minimumReleaseAgeStrict: false` も入れています。
これは、条件に合う古い版があるときはそちらを優先しつつ、互換範囲の中に成熟版がないときだけインストール自体は続ける設定です。
学習用リポジトリとして「安全性を高めつつ、セットアップ不能にはしにくい」バランスを取っています。

また、`minimumReleaseAgeExclude` には一部のパッケージがバージョン付きで入っています。
これは今の教材構成で必要な依存のうち、まだ 14 日待機に達していない版だけを一時的に明示したものです。
広く `*` で除外するのではなく、必要な版だけを絞って例外にすることで、待機ルールをできるだけ維持しています。

もちろん、急いで取り込みたい修正があることもあります。
そのときは理由を確認したうえで更新する、という判断をチームでできる状態が大切です。

## 3. `pnpm audit` で既知の脆弱性を調べる

この教材では、ローカルでも CI でも次を実行できます。

```bash
pnpm run audit
```

これは、利用中の依存関係に「既知の脆弱性」が報告されていないかを確認するコマンドです。
CI では `high` 以上の severity を失敗条件にしています。

これを入れる理由は、アプリのコードが正しくても、依存関係に危険な既知問題があると公開物全体の安全性が下がるからです。

## 4. Dependabot にも待機時間を入れる

この教材では `.github/dependabot.yml` に `cooldown` を設定しています。

考え方は `minimumReleaseAge` と似ています。

- `minimumReleaseAge`: ローカルや CI で新しい版をすぐ採用しない
- `cooldown`: 更新 PR をすぐ大量に出しすぎない

これにより、更新のたびに落ち着いて中身を確認しやすくなります。
依存関係の更新は自動化しつつ、「レビューしてから入れる」流れを保ちやすくするのが狙いです。

## 5. `SBOM` を生成する

`SBOM` は `Software Bill of Materials` の略です。
やさしく言うと、「このアプリにどんな OSS が入っているか」の一覧表です。

この教材では次で生成できます。

```bash
pnpm run sbom
```

生成される `sbom.spdx.json` は、使っているパッケージの棚卸しに役立ちます。

- 後から脆弱性調査をするとき
- 監査や説明が必要なとき
- 利用中 OSS を整理したいとき

CI ではこの SBOM を artifact として保存するので、実行結果をあとから確認できます。

## 6. ビルドスクリプトを明示的に許可する

一部のパッケージは、インストール中に追加のビルドスクリプトを実行します。
便利な反面、ここは supply chain attack の入口にもなりやすい場所です。

この教材では `pnpm-workspace.yaml` の `allowBuilds` で、`esbuild` だけを明示的に許可しています。
`esbuild` は `Vite` 系ツールの動作に必要ですが、それ以外までまとめて許可すると、何が動いているか見えにくくなります。

つまり方針は次の通りです。

- 必要なビルドスクリプトだけを許可する
- 何を許可したか設定ファイルに残す
- 後からレビューで見返せるようにする

## 7. PR では何を見るか

依存関係を変える PR では、次をセットで確認してください。

- `package.json`
- `pnpm-lock.yaml`
- CI の `audit` 結果
- SBOM が生成できているか

特に `package.json` だけ変わっていて lockfile が更新されていないと、ローカルと CI で入る依存がずれることがあります。

## まとめ

この教材では、依存関係の安全管理を次の形で実践しています。

- `pnpm-lock.yaml` でバージョンを固定する
- `minimumReleaseAge` で新しすぎる版を少し待つ
- `pnpm run audit` で既知の脆弱性を確認する
- `Dependabot cooldown` で更新 PR の出方も調整する
- `allowBuilds` で必要な install script だけを許可する
- `pnpm run sbom` で部品表を残す

CI/CD は「アプリを自動で公開する仕組み」だけではありません。
「安全に公開できる状態かを自動で確かめる仕組み」でもある、という点もこの教材で体験できます。
